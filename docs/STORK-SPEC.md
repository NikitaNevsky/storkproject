## КОНТЕКСТ ПРОЕКТА

**Stork (Аист)** — гипер-локальный сервис доставки свежих продуктов и собственной выпечки в формате «как Самокат, но мультиформат». Российский клиент строит сеть точек на территории своих ТЦ; каждая точка имеет:
- собственный каталог и склад
- закрепленные зоны доставки (полигоны по районам)
- курьеров (свои, не аутсорс)

**Ключевые отличия от Самоката:**
- **Слотовая доставка** (как Сбермаркет), а не ASAP по 15 минут
- **Самовывоз** как первоклассная опция (точка = магазин в ТЦ)
- **Собственная пекарня и готовая еда** — главный USP, не просто продукты

**Регион запуска:** Москва + МО. После релиза — масштабирование на другие города.

**Стадия:** клиенту согласован HTML-прототип (52 экрана клиентского флоу, 8 экранов курьера, 9 разделов админ-панели, лендинг-сайт). HTML-прототипы — это design source of truth. Поведение, контент, бренд — оттуда.

- **Cart badge** — оранжевый кружок `#F7982B` с числом графитом, white border 2px, top:-5px right:-9px на иконке.
- **Bottom nav (mobile)** — 5 табов (Главная / Каталог / Корзина·badge / Бонусы / Профиль), иконки 24px, текст 10px weight 700, активный таб оранжевый.

## ТЕХНИЧЕСКИЙ СТЭК (зафиксирован)

### Mobile (iOS + Android)
- **Expo SDK 51+** (managed workflow, EAS Build для прода)
- **React Native + TypeScript**
- **Reanimated 3** (для FAB-корзины с летящими товарами и других микро-анимаций)
- **Gesture Handler** (для свайпов в сторис, drag в корзине)
- **Expo Router** (file-based routing, group routes для табов)
- **Zustand** (state, ничего тяжёлого типа Redux)
- **TanStack Query** (server state, кэш, оптимистические апдейты)
- **React Hook Form + Zod** (формы и валидация)
- **Expo Notifications** (push через FCM/APNs)
- **Expo Location** (определение районов доставки)
- **React Native MMKV** (быстрый локальный сторадж вместо AsyncStorage)
- **expo-image** (нативный image-loader с кэшем)

### Web / Admin
- **Next.js 15** (App Router, RSC где имеет смысл)
- **TypeScript**
- **Tailwind CSS** + дизайн-токены из палитры выше
- **shadcn/ui** (Radix-based примитивы, не зависимости — копируется в проект)
- **TanStack Table** (для админских таблиц заказов/пользователей)
- **react-yandex-maps** (карты, не Google — Россия)

### Backend
- **NestJS** (модульная архитектура, удобно для команды)
- **TypeScript строго**
- **PostgreSQL 16** + **Prisma ORM**
- **PostGIS** (расширение Postgres для геополигонов районов)
- **Redis** (кэш каталога per-store, очереди заказов, rate-limiting)
- **BullMQ** (фоновые задачи: пересчёт уровней лояльности, отправка пушей)
- **Socket.io** (realtime трекинг курьера + чат поддержки)
- **MinIO / S3** (хранение фото товаров)

### Платежи (РФ)
- **СБП через Тинькофф Эквайринг** или **CloudPayments**
- **SberPay** (для пользователей Сбера)
- Картой онлайн (через того же эквайера)
- Картой курьеру / наличные (без интеграции, помечается флагом в админке)

### Инфра
- Docker Compose для dev
- GitHub Actions для CI
- Деплой: VPS (Yandex Cloud / Selectel) с docker-compose в проде на первом этапе. После 1000 заказов/день — миграция на k8s.

## АРХИТЕКТУРА

### Монорепо (Turborepo или pnpm workspaces)
```
/apps
  /mobile          ← Expo + RN (клиент + курьер, разделение по env)
  /admin           ← Next.js (админка для управления сетью)
  /web             ← Next.js (веб-витрина для покупателей)
  /api             ← NestJS
/packages
  /shared-types    ← типы DTO между фронтом и бэком
  /ui-tokens       ← цвета, типографика как JSON (синхронизация Tailwind ⇄ RN StyleSheet)
  /icons           ← общий пак Tabler-иконок
  /eslint-config
  /tsconfig
```

### Multi-store модель
- `Store` (точка) → имеет `Polygon` (PostGIS, геозона доставки) → принимает заказы только из этой зоны
- `Product` (мастер-каталог сети) → `StoreInventory` (per-store наличие и цена)
- При определении адреса юзера: PostGIS-запрос `ST_Contains(polygon, userPoint)` → находим Store → грузим её каталог из Redis-кэша
- Если адрес вне всех зон — экран «Stork пока не доставляет в ваш район»

### Заказ
```
DRAFT → CREATED → ACCEPTED → COLLECTING → READY_AT_STORE 
      → COURIER_ASSIGNED → IN_TRANSIT → DELIVERED
      ↘ CANCELLED (с любого шага до IN_TRANSIT)
```

### Слоты доставки
- Каждая Store имеет расписание слотов (например, 8:00-10:00, 10:00-12:00, ...) с лимитом одновременных заказов
- При чекауте мы показываем только доступные слоты
- Cron-job каждые 5 минут пересчитывает доступность

  - Аналитика (заглушка)

**Deliverable:** покупатель может оформить заказ с реальной оплатой через СБП.

### Фаза 4 · Realtime + Courier app + Chat (3-4 недели)

**Deliverable:** запуск с одной точкой в Хамовниках.

## ПРИНЦИПЫ РАЗРАБОТКИ

1. **HTML-прототип — это truth.** Не упрощать визуал «потому что в Tailwind так не сделать». Если нужен кастомный CSS — делай.
2. **Mobile-first** во всём. Десктоп — вторичен.
3. **Server-driven contentful** где можно: тексты, баннеры, FAQ — через CMS-таблицу в админке, не хардкод.
4. **Один источник дизайн-токенов** — `/packages/ui-tokens`. Если нужен новый цвет — добавляй туда и используй в обоих стэках.
5. **TypeScript strict mode**, никаких `any`. DTO между фронтом и бэком — общие типы из shared-types.
6. **Никаких `console.log` в проде**, только logger (Pino на бэке, простая обёртка на фронте).
7. **Все async-операции через TanStack Query на фронте**, не useEffect+fetch.
8. **Оптимистические апдейты в корзине** (мгновенный отклик, фоновая синхронизация).
9. **Тесты пишем минимально, но критическое покрываем:** оформление заказа (e2e), расчёт корзины с бонусами (unit), геозоны (unit).
10. **Production-grade с самого начала:** Sentry, structured logging, health-checks, graceful shutdown.

## ПЕРВЫЕ ШАГИ (что делать после прочтения)

   - Mobile-only или одновременно с веб-витриной?
   - Размер команды разработки и желаемые сроки?

3. Подними Docker Compose локально, проверь что Postgres + PostGIS + Redis работают.
4. Распиши задачи на Фазу 0 в виде ToDo и согласуй со мной.
